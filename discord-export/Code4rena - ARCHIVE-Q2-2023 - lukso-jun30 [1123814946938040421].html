CJ42 07/04/2023 07:55
Wardens, if you come across functions that look like this and wonder why we made the function payable but reverted if some value is sent to it, here are the rationale. function setData( bytes32 dataKey, bytes memory dataValue ) public payable virtual override onlyOwner { if (msg.value != 0) revert ERC725Y_MsgValueDisallowed(); _setData(dataKey, dataValue); } In Solidity, you cannot override a function as follow:
  • neither override from payable -➝ nonpayable ❌
  • nor from nonpayable -➝ payable ❌ We want to enforce that no value can be sent to this function
But allow anyone to override the behaviour if they want this function to allow accepting value. Since we cannot override from nonpayable —> payable, we decided to make the function always payable (to allow overriding this behaviour) but revert by default if value is sent (to make it feels like if it is nonpayable) Asked by @wangxx2026 but thought sharing it as well for anyone else so that you get the context.

CJ42 07/04/2023 08:42
The magic value MUST start with 0x9bf04b as its first 3 bytes. case 1: if the magic value starts with these first 3 bytes 0x9bf04b, that means the verification succeeded, there is nothing else that needs to be done. The function can be executed (e.g: ERC725Y setData(bytes32,bytes)). case 2: if the magic value is 0x9bf04b01 (with these first 3 bytes + the 4th byte is strictly 01), that means the verification succeeded, but the contract has to do a post verification because the function being run is making an external call (e.g: ERC725X execute(uint256,address,uint256,bytes)). This is necessary to prevent scenario of reentrancy in particular.
Note: for case1, we don't care what the 4th byte is, as long as it is not 01
case 3: if the value being returned does not start with 0x9bf04b, it is an invalid magic value, so we revert with the custom error LSP20InvalidMagicValue // bytes4(bytes.concat(bytes3(ILSP20.lsp20VerifyCall.selector), hex"01")) bytes4 constant _LSP20_VERIFY_CALL_MAGIC_VALUE_WITH_POST_VERIFICATION = 0x9bf04b01; // bytes4(bytes.concat(bytes3(ILSP20.lsp20VerifyCall.selector), hex"00")) bytes4 constant _LSP20_VERIFY_CALL_MAGIC_VALUE_WITHOUT_POST_VERIFICATION = 0x9bf04b00; (edited)

0xYamen 07/04/2023 09:04
LSP0 have an owner KeyManager X Lets say EOA A, called setData function, the setData function will identify that the caller is the not the owner.   The setData function will use the internal _verifyCall function to call the lsp20VerifyCall(..) function on the owner passing the caller (EOA A), value (0 let’s say), the calldata (setting 2 data keys let’s say).   The role of the lsp20VerifyCall(..) function is to have a logic that verify wether the call is allowed based on the parameter.    In the case of the KeyManager, we well check the permission of the caller (EOA A), if he has permission to set these data keys (taken from calldata), then the function will return the magic value if the caller is allowed.   The lsp20VerifyCallResult in the KeyManager serve to re-set back to the Reentrnacy guard to false.   When a setData is done, we don’t set the Reentrnacy guard to true because setData cannot reenter, that’s why we don’t need to trigger the lsp20VerifyCallResult in case of setData. So we return the magic value (3 bytes with whatever byte at the end but not 0x01) When an execute is done, we set the Reentrnacy guard to true because execute can reenter, that’s why we need to trigger the lsp20VerifyCallResult in case of execute. So we return the magic value (3 bytes with0x01 as the last byte indicating that we need to trigger lsp20VerifyCallResult to re-set the reentrancy guard to false)  
in case of execute, only the first execute should trigger the lsp20VerifyCallResult function to re-set the reentrancy guard to false, as in case of chaining several execute functions together, you don’t want the second one to re-set the reentrancy guard to false in the middle and make the subsequent call not require REENTRACNY Permission.  Because REENTRACNY permission is only required when the Reentrnacy guard is set to true.
09:06
------------------------------------------------------------------------------------------------ When an LSP0 have an owner such as EOA, or a contract that does not implement the lsp20VerifyCall function or a contract that implement the fallback. When an EOA X calls a function of the LSP0, this function will see that the caller is not the owner, so it will call the lsp20VerifyCall function on the owner. In case of all these owners mentioned above, none of them will return the magic value, so the call verification be unsuccessful and the call will revert (edited)
